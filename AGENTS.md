# AGENTS.md

本项目是操作系统课程的电梯调度系统，目标不是单纯生成一个能跑的程序，而是在 Agent 辅助下边开发边学习 Go 后端、HTTP API、前端交互、调度算法和基本工程 workflow。

后续 Agent 开始工作前，应先阅读本文，并按需要查阅 `docs/` 下对应文档。

## 文档索引

### `docs/first-analysis.md`

项目需求来源和课程要求说明。

Agent 应从这里确认：

- 系统基本规模：默认 20 层、5 部电梯，但设计上应尽量可配置。
- 并发要求：最终应体现“每部电梯作为独立执行单元”的思想。
- 功能范围：电梯内部按钮、楼层上下行按钮、状态显示、请求联动、停靠开门等。
- 调度要求：至少实现一种经典或可解释的调度算法，并支持多算法切换。
- 界面要求：需要基础可视化界面，显示楼层、电梯方向和请求按钮。
- 提交材料：设计方案报告、源代码、可执行程序。

### `docs/goals.md`

用户的学习目标和协作偏好。

Agent 应重点遵守：

- 不做纯粹 Vibe Coding，不直接交付大段黑盒代码。
- 重要设计先解释，再实现；实现后再带用户读关键文件。
- 代码结构、命名和注释要适合初学者阅读。
- Go 后端、HTTP API、前后端通信、goroutine、channel、调度算法、项目的构建如 `npm` 等是**重点学习内容**。
- 保留用户自己动手实现和修改的空间，尤其是算法核心、接口设计、状态结构体等部分。
- 对调度算法、并发模型等项目核心部分，不应以“学习版”“简化版”“更容易写对”为最终目标；应先分析更高质量、更接近真实系统的设计，再把实现拆成可学习的小步骤推进。
- 每个阶段要有明确交付物，便于用户知道当前完成了什么、学到了什么。
- 最终报告材料要在开发过程中逐步沉淀，而不是最后临时拼接。

### `docs/instructions-from-agent.md`

当前推荐开发路线和阶段 checklist。

Agent 应从这里确认下一步任务。除非用户明确改变方向，否则按该 checklist 的顺序推进：

1. 最小后端程序
2. 电梯核心数据模型
3. 同步版本系统逻辑
4. 最小 HTTP API
5. 最小前端页面
6. 调度算法
7. 并发模型
8. 测试和调试
9. 界面改善
10. 课程提交材料

推进原则：

- 每次只做一个小目标。
- 不把模型、API、前端、并发混在一次改动里。
- 先让代码能运行，再追求更漂亮的设计。
- 完成阶段任务后，更新 checklist 中对应项。

### `docs/record.md` 和 `docs/record2.md`

项目开发历程和学习笔记。

`docs/record.md` 是早期学习记录，内容较长，主要作为历史资料查阅。

从现在开始，新的开发记录和学习解释统一写入：

```text
docs/record2.md
```

Agent 应使用 `docs/record2.md` 记录：

- 每个阶段新增了什么文件和能力。
- 如何运行、如何验证。
- 关键代码阅读顺序。
- Go、HTTP、前后端通信、调度算法、并发等概念解释。
- 遇到的问题、解决方式和环境注意事项。
- 可复用到最终报告里的设计思路。

如果用户提出“这里我不懂”“解释一下某段代码”“这个概念是什么”，优先把新的解释写入 `docs/record2.md`，而不是继续追加到 `docs/record.md`。

除非用户明确要求“修改前文”“写入对应部分”或修正已有内容，否则新增开发记录和学习解释必须追加到 `docs/record2.md` 文件末尾，保持时间顺序，不要插入到中间旧段落。

### `docs/git提交规范.md`

提交习惯说明。

Agent 应遵守：

- 提交信息使用 `feat`、`docs`、`chore`、`test` 等前缀。
- 每次提交尽量是一个最小改动。
- 不把无关改动混到同一次提交。

如果用户要求提交代码，提交前应先查看 `git status`，并避免提交不属于本次任务的无关改动。

## 当前项目结构约定

```text
cmd/server/          Go 程序入口
internal/api/        HTTP API handler
internal/elevator/   电梯系统核心模型、调度和运行逻辑
web/                 前端页面、样式和 JavaScript
docs/                需求、目标、学习记录和阶段指令
```

`main.go` 只负责启动服务、创建系统和注册路由。

## 技术栈约定

- 后端使用 Go。
- 后端优先使用标准库，当前 HTTP 服务使用 `net/http`。
- 前端采用稳妥学习路线：先用原生 HTML、CSS、JavaScript 做一个极小的通信调试页，打通 `GET /api/state` 和 `POST /api/request`；闭环跑通后尽快迁移到 Vue + Vite 作为正式前端。
- 原生前端阶段只验证通信和最小展示，不做复杂动画、组件拆分、算法切换 UI 或大量 DOM 工程。
- Vue 阶段先保持极简：JavaScript，不用 TypeScript；单页面，不用路由；原生 CSS，不引入组件库；先从 `App.vue` 和少量组件开始。
- 初期先实现同步、可读、可测试的模型；并发版本在同步版本稳定后再加入。

## Agent 协作方式

- 动手前先读相关代码和文档，不凭空假设项目状态。
- 做代码改动前，先简短说明将改哪些文件、为什么这样改。
- 对初学者不熟悉的概念，要结合当前代码解释。
- 不直接生成超大块复杂代码；优先小步推进。
- 小步推进不等于降低最终设计目标。涉及核心算法时，应先说明目标设计和取舍，再选择一个能通向该目标的增量实现路径。
- 生成代码后，应说明核心文件、关键函数和验证方式。
- 重要学习解释写入 `docs/record2.md`。
- 阶段推进状态写入 `docs/instructions-from-agent.md`。
- 不随意重构目录或引入新依赖，除非能明确降低复杂度或符合当前阶段目标。
- 所有代码需要有合适的注释，**尤其是函数的功能的注释，参考已有的实现**。

## 运行和验证

当前最小后端可用：

```bash
go run ./cmd/server
```

健康检查：

```bash
curl http://localhost:8080/api/health
```

基础检查：

```bash
go test ./...
```

后续每次修改 Go 代码后，应至少运行：

```bash
gofmt -w <changed-go-files>
go test ./...
```

如果当前环境因沙箱限制无法运行命令，应在最终说明中明确写出未能验证的原因。

## 提交规范

提交类型参考：

- `feat`: 新功能，例如新增 API、调度算法、前端交互。
- `docs`: 文档更新，例如学习记录、设计说明、报告材料。
- `chore`: 工程性改动，例如初始化目录、调整配置。
- `test`: 测试相关改动。

保持小提交。一次提交应能用一句话解释清楚。
