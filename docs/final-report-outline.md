## 4. 调度算法

> 这是报告的核心章节。五个算法都列出，但对 FCFS / First-Available / Nearest-Idle 只做简短说明；**SCAN 和 LOOK 需要详细展开。**

### 4.1 统一调度接口

- `Scheduler` 接口：`Name() string` + `Assign(*System) bool`
- 为什么用接口：方便 `SetScheduler(name)` 运行时切换，前端下拉切换即生效

### 4.2 简单算法（简略）

- **First-Available**：第一条可用电梯接最早的请求。说明它的局限（不考虑距离，不适合多电梯）
- **FCFS（先来先服务）**：按请求创建顺序分配。说明公平性（FCFS 公平）与效率（可能造成长等待）的权衡
- **Nearest-Idle**：最早 pending 请求分配给距离最近的空闲电梯。说明它比 First-Available 更合理（考虑了空间距离）

### 4.3 SCAN 算法（重点）

- **算法来源**：磁盘调度中的电梯算法（Elevator Algorithm / SCAN）
- **核心思想**：
  - 每部电梯维护长期扫描方向 `ScanDirection`
  - 沿当前方向持续服务，走到没有更多请求时反向
  - 顺路的运行中电梯可以追加新请求（不只是空闲电梯才接单）
- **执行过程**：
  - 枚举所有 pending 请求 × 所有电梯
  - 过滤：空闲电梯都可选；运行中电梯需判断是否顺路
  - 对每个可行组合用 cost 函数评分
  - 选 cost 最低的组合执行分配
- **Stops 排序规则**：上行按楼层升序，下行按楼层降序
- **关键判断函数**：`canAppendSCANRequest`（顺路判断）、`isFloorAheadInSCAN`（请求是否在扫描前方）、`requestMatchesSCANDirection`（hall 请求方向需匹配扫描方向）
- **举例**：给一个具体场景，展示 SCAN 如何选择候选

### 4.4 LOOK 算法（重点）

- **与 SCAN 的区别**：SCAN 走到物理边界（1 楼 / 顶层），LOOK 只走到当前方向上最后一个请求处就反向
- **现实对应**：现实中电梯不会空跑到顶楼再回头——LOOK 更贴近真实
- **如何实现**：在 SCAN 的基础上修改反向条件——扫描方向上没有更多 pending 请求时即反向，不需要走到边界
- **性能对比**：和 SCAN 相比，LOOK 减少无效移动

### 4.5 cost 函数

- 这是调度器的决策核心
- **结构**：`DistanceCost + TurnPenalty + StopPenalty - WaitCompensation`
- 四个维度的含义和公式
- 为什么要有等待补偿：避免长期饥饿
- cost 与 SCAN/LOOK 的关系：cost 是"选哪部电梯"，SCAN/LOOK 是"这部电梯内部如何排序"

## 5. Go 并发模型

> 对应"学习特定环境下多线程编程方法"的学习目标。
> 这个章节可以兼作 Go 并发概念的介绍，不需要假设读者已经懂 goroutine。

### 5.1 为什么要并发

- 课程要求"每部电梯作为独立执行单元"
- 现实对应：多部电梯确实并行运行
- 本项目的并发需求：API handler、自动 ticker、每部电梯的独立运行

### 5.2 Go 并发基础概念（简短）

- goroutine：比线程更轻量的并发执行单元，`go f()` 启动
- channel：goroutine 之间传递消息的管道，`ch <- v` 发送，`v := <-ch` 接收
- select：同时等待多个 channel
- mutex：互斥锁，保护共享数据。Go 的 `sync.Mutex` 不可重入
- context：控制 goroutine 生命周期，`ctx.Done()` 通知退出
- **和传统多线程的对比**：Goroutine vs OS Thread（不要求详细，点到关键区别即可）

### 5.3 本项目的并发设计

- **两把锁的职责分离**：
  - `mu` — 保护共享数据（Elevators、Requests、CurrentTick）
  - `stepMu` — 保护 tick 边界（同一时间只有一个 Step 在执行）
  - 为什么需要两把锁：Step 中间释放 mu 让 API 和电梯并行，但不允许两个 Step 交错
- **电梯 goroutine 的通信模式**：
  - Step → elevatorTickCommand（含状态副本）→ 每部电梯 goroutine
  - 电梯 goroutine → elevatorTickResult（新状态 + 完成的请求 ID）→ Step
- **核心安全原则**：goroutine 不直接写共享 System；共享状态写入集中在 Step 的合并阶段
- **为什么做 `cloneElevator`**：Go 切片共享底层数组，并发时必须深拷贝，否则数据竞争
- **一张图**：完整的 tick 执行流程（调度 → 分发 → 并行计算 → 合并 → tick++）
