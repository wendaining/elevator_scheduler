摘要

本项目实现了一个面向操作系统课程的电梯调度算法可视化系统。系统以多部电梯共享楼层请求为基本场景，将乘客请求类比为待调度任务，将电梯类比为资源执行单元，由调度器决定请求分配，并由每部电梯的独立 goroutine 推进运行状态。后端使用 Go 标准库 net/http 提供 API，使用 SQLite 保存已完成请求；前端使用 Vue 3、Element Plus 与 Vite 实现可交互的楼层请求、轿厢请求、算法切换和运行日志展示。

关键词：电梯调度；操作系统；Go；goroutine；HTTP API；Vue；SQLite

1. 项目概述

1.1 项目背景

电梯调度是现实生活中的一个经典问题，同时也是操作系统调度问题的一个直观类比：多部电梯对应多个可执行资源，楼层请求和轿厢内请求对应等待被处理的任务，调度算法需要在公平性、响应速度和资源利用率之间作出权衡。本项目以一栋默认 20 层、5 部电梯的大楼为基础场景，但在实现上将楼层数、电梯数和运行节奏设计为可配置参数，便于测试不同规模下的调度行为。

本项目实现电梯调度过程与算法的可视化，提供 Web 可视化界面，支持在前端直接发起模拟电梯调度的请求、紧急制动和调度算法切换，后端持续推进离散时间片，并将当前系统状态通过 HTTP API 暴露给前端。

1.2 项目目标

  • 实现一个可运行的多电梯调度系统，支持楼层外部请求、轿厢内部请求和电梯状态展示。
  • 实现并切换多种调度算法，包括第一可用（First Available）、最近空闲（Nearest Idle）、先来先服务（FCFS）、扫描算法（SCAN）和 LOOK 扫描。
  • 使用 Go 的 goroutine 和 channel 表达“每部电梯作为独立执行单元”的并发模型。
  • 通过 HTTP API 和 Vue 前端完成前后端通信，并将调度过程可视化。
  • 使用 SQLite 保存已完成请求，为后续算法统计和实验对比提供数据来源。

1.3 系统运行流程

系统启动后，后端创建 System 实例和电梯运行 goroutine，并自动周期性调用 Step() 推进模拟时间。前端页面通过轮询 GET /api/state 获取系统快照，用户在界面中点击楼层按钮或提交轿厢请求后，前端向后端发送 HTTP 请求，后端将请求写入运行态数据结构，再由当前调度器在后续 tick 中完成分配。

2. 系统设计与实现

2.1 系统架构概览

系统采用前后端分离但部署简单的结构。前端负责渲染大楼、电梯、请求按钮和配置面板；后端负责维护状态，所有请求创建、调度算法切换、系统重启和紧急制动均通过 API 进入后端。调度层不直接处理 HTTP，而是围绕 System、Request、Elevator 和 Scheduler 等核心类型实现业务逻辑。

层次 主要文件或目录 职责
程序入口 cmd/server/main.go 创建系统配置、初始化电梯系统、启动自动 tick、注册 HTTP 路由。
API 层 internal/api/handler.go 接收前端请求，完成 JSON 解码、输入校验、系统调用和响应编码。
核心模型 internal/elevator/model.go、system.go 定义电梯、请求、停靠计划和系统状态，并提供状态推进方法。
调度层 internal/elevator/*Scheduler.go 实现不同调度策略，统一通过 Scheduler 接口调用。
电梯运行层 internal/elevator/elevator_runner.go 每部电梯使用独立 goroutine 接收 tick 命令并计算下一状态。
持久化层 internal/elevator/request_store.go 将完成请求写入 SQLite，供后续统计和算法对比使用。
前端 web/src 实现状态轮询、请求提交、控制面板、日志终端和可视化布局。

调度算法和电梯运行逻辑都位于后端核心模型中，前端只通过 API 操作系统状态，不复制调度逻辑；SQLite 只记录历史数据，不影响运行态调度性能。

2.2 项目结构

os_sp26_proj1/

├── web/                 # Vue 3 + Element Plus + Vite 前端

├── analysis/            # 调度实验与图表输出
├── scripts/             # 本地分析脚本

Go 项目中 internal 目录表示内部包，不希望被仓库外部直接导入。该结构使 API 层和电梯核心模型在同一后端项目中保持清晰边界：internal/api 关心 HTTP 输入输出，internal/elevator 关心调度和状态推进。

2.3 核心数据模型

系统的核心数据模型围绕请求、停靠计划、电梯和系统四类对象展开：

类型 关键字段 设计说明
Direction idle、up、down 统一描述电梯运动方向和请求方向。
Request ID、Floor、Kind、Status 表示 hall 或 cabin 请求，状态包括 pending、assigned、done。
StopPlan Floor、Reason、RequestIDs 描述一次停靠及其原因，同一停靠可完成多个请求。
Elevator CurrentFloor、Direction、Stops 保存单部电梯当前状态、后续停靠计划和门/移动倒计时。
System Elevators、Requests、CurrentTick 保存全局运行态状态，是调度器和 API 共同操作的核心对象。

先说明系统中重要的字段和概念：hall 请求表示乘客在楼层外按上行或下行按钮，例如“14 楼上行”；它还没有天然归属某部电梯，必须由调度器决定哪部电梯响应。cabin 请求表示乘客进入某部电梯后在轿厢内选择目标楼层，例如“Cabin #4 去 11 楼”；它天然归属已选中的电梯，因此创建时会携带 ElevatorID 并直接加入这部电梯的停靠计划。请求状态中的 pending 表示已创建但尚未分配，assigned 表示已被某部电梯接收并写入停靠计划，done 表示请求已在开门完成时结束并写入历史数据库。

字段 / 概念 含义 在系统中的作用
Kind 请求来源，取值为 hall 或 cabin。 决定请求是否需要调度器分配；hall 请求等待调度，cabin 请求直接归属指定电梯。
Direction 即时方向，表示电梯当前正在上行、下行或空闲。 用于前端展示和单 tick 状态推进。
ScanDirection 调度方向，表示 SCAN / LOOK 算法当前优先服务的长期方向。 用于判断请求是否顺路、停靠计划如何排序，和即时运动方向不完全等同。
StopPlan 一条停靠计划，包含楼层、停靠原因、方向和关联请求 ID。 把抽象请求转换成电梯可以执行的动作，并允许同层多个请求合并完成。
RequestIDs 某次停靠对应的一组请求编号。 开门完成时据此批量标记请求完成，避免同层 hall 与 cabin 请求互相丢失。
CurrentTick 全局模拟时间。 统一计算等待时间、移动时间、开门时间、报警暂停时间和算法统计指标。

从设计结构看，核心模型是一套围绕“请求如何被创建、分配、执行、完成”的状态体系。围绕这套结构，几个关键设计点分别解决不同层次的问题：

设计点 解决的问题 实现方式
运行态请求表 请求不能在分配后从系统中消失，否则无法记录等待时间、分配电梯和完成时间。 Requests map[int64]*Request 保存 pending 和 assigned 请求；完成后写入 SQLite 并从 map 删除。
hall / cabin 请求区分 楼层外部请求需要调度，电梯内部请求天然属于某部电梯，不能被分给其他电梯。 cabin 请求携带 ElevatorID，在 AddRequest 中直接分配给对应电梯，不进入 pending。
停靠计划 只用楼层整数无法区分同层上行、同层下行和轿厢目标，容易错误合并停靠。 StopPlan 记录 Floor、Reason、Direction 和 RequestIDs。
离散 tick 时间 调度、移动、开门、请求等待都需要统一时间单位，直接使用真实时间不利于测试。 CurrentTick 记录全局模拟时间，自动 ticker 每隔固定真实时间调用一次 Step()，每次 Step 推进一个模拟 tick。
电梯状态机 电梯不只是“在哪一层”，还要描述移动、开门、扫描方向和异常暂停。 Elevator 保存方向、倒计时、停靠计划和 EmergencyStop；紧急制动只是状态机中的一种暂停状态。

tick 是本系统的统一时间单位。后端默认用真实时间中的固定间隔触发一次自动步进，步进时调用 System.Step()，调度器先分配请求，再让电梯状态前进一步，最后把 CurrentTick 加一。这样设计的好处是测试可以直接调用 Step() 推进任意数量的 tick，不依赖真实计时器，也能精确比较不同算法的等待时间。

tick 相关字段 含义 示例用途
CreatedTick 请求被创建时的 tick。 用于计算请求已经等待了多久。
AssignedTick 请求被分配给电梯时的 tick。 用于区分调度等待和电梯执行耗时。
CompletedTick 请求完成并写入历史记录时的 tick。 用于统计总耗时和生成算法对比数据。
TicksPerFloor 电梯移动一层需要的 tick 数。 把楼层距离转换成模拟运行时间。
DoorBaseTicks、TickPerPassenger 开门基础时间和乘客上下电梯追加时间。 让停靠不只是瞬间完成，而会占用若干 tick。
EmergencyRemainingTicks 紧急制动剩余暂停 tick 数。 报警触发后让对应电梯暂停固定时间，再自动恢复。

2.4 HTTP API 设计

后端 API 采用 JSON 作为请求和响应格式。所有会修改系统状态的接口使用 POST，只读接口使用 GET。handler 层会拒绝不匹配的方法、非法 JSON、未知字段和不合法业务参数。

方法 路径 功能
GET /api/health 健康检查，返回服务是否可用。
GET /api/config 返回前端轮询和展示所需配置，如自动 tick 间隔、移动耗时、开门耗时和紧急制动 tick 数。
GET /api/state 返回完整系统快照，包括楼层数、电梯状态、运行态请求和当前调度算法。
POST /api/request 创建 hall 或 cabin 请求。
POST /api/scheduler 切换调度算法，并重建系统运行周期。
POST /api/floor-count 调整楼层数，重启系统。
POST /api/elevator-count 调整电梯数，重启系统。
POST /api/elevator-emergency 触发指定电梯的紧急制动。

2.4.1 请求示例

创建楼层外部上行请求：

POST /api/request
Content-Type: application/json

{
  "floor": 6,
  "direction": "up",
  "kind": "hall"
}

创建 3 号电梯内部到 12 楼的请求：

POST /api/request
Content-Type: application/json

{
  "floor": 12,
  "direction": "idle",
  "kind": "cabin",
  "elevatorId": 3
}

触发 4 号电梯紧急制动：

POST /api/elevator-emergency
Content-Type: application/json

{
  "elevatorId": 4
}
错误处理约定：非法输入返回 400，方法不匹配返回 405，服务器内部状态或编码错误返回 500。JSON 解码使用 DisallowUnknownFields，避免前端提交未定义字段后被后端静默忽略。

2.5 SQLite 持久化

系统使用 SQLite 保存已完成请求。数据库表 completed_requests 的字段与 Request 结构基本对应，包括请求 ID、楼层、方向、请求来源、状态、创建 tick、分配 tick、完成 tick 和响应电梯 ID。请求完成时，后端调用 RequestStore.SaveCompletedRequest 写入数据库。

CREATE TABLE IF NOT EXISTS completed_requests (
  id INTEGER PRIMARY KEY,
  floor INTEGER NOT NULL,
  direction TEXT NOT NULL,
  kind TEXT NOT NULL,
  status TEXT NOT NULL,
  created_tick INTEGER NOT NULL,
  assigned_tick INTEGER NOT NULL,
  completed_tick INTEGER NOT NULL,
  assigned_elevator_id INTEGER NOT NULL
);

将运行态请求和历史请求分离，方便完成记录被 Python 脚本或 SQL 查询读取，用于后续绘制算法对比图和统计完成时间。

2.6 前端实现

前端使用 Vue 3 构建单页应用，核心状态来自后端 /api/state 的轮询结果。页面主体由大楼视图、右侧控制面板和日志终端组成。大楼视图展示各电梯当前所在楼层、方向和楼层请求按钮；控制面板负责系统配置、调度算法切换、cabin 请求和紧急制动；日志终端记录用户操作与请求完成事件。

  • 楼层上/下行按钮：提交 hall 请求。
  • 选中电梯后设置目标楼层：提交 cabin 请求。
  • 调度算法下拉框：调用 /api/scheduler 切换算法。

3. 调度算法

调度算法是本项目最核心的部分。系统把乘客请求抽象成 Request，把每部电梯抽象成可执行资源，调度器负责在每个 tick 开始时从所有等待中（pending）的请求里挑选一个可行分配，并把请求转换成对应电梯的 StopPlan。当前系统实现了 First Available（第一可用）、FCFS（先来先服务）、Nearest Idle（最近空闲）、SCAN（扫描/电梯算法）和 LOOK（LOOK 扫描）五种调度策略。

3.1 统一调度接口

所有调度器都实现同一个 Scheduler 接口。接口只有两个方法：Name() 返回算法名称，Assign(*System) bool 执行一次分配尝试。这样 System.Step() 不需要知道当前使用的是哪一种具体算法，只需要调用 s.scheduler.Assign(s)。

type Scheduler interface {
    Name() string
    Assign(system *System) bool
}

NewScheduler(name) 是调度器工厂函数，根据前端传入的名称创建具体调度器。前端通过 POST /api/scheduler 切换算法；后端的处理方式是重建一个新的 System，并为新的运行周期创建独立 SQLite 数据库文件。这保证同一个数据库文件只对应一种调度算法，便于后续统计分析。

算法名称 中文说明 前端/API 名称 实现文件
First Available 第一可用 first-available internal/elevator/FirstAvailableScheduler.go
FCFS 先来先服务 fcfs internal/elevator/FCFSScheduler.go
Nearest Idle 最近空闲电梯优先 nearest-idle internal/elevator/NearestIdleScheduler.go
SCAN 扫描算法 / 电梯算法 scan internal/elevator/SCANScheduler.go
LOOK LOOK 扫描算法 look internal/elevator/LOOKScheduler.go

3.2 简单算法

简单算法主要用于建立调度接口、验证请求流转和作为高级算法的对照组。它们的共同特点是只处理 pending 请求中最早创建的一条请求，也就是通过 firstPendingRequestID 取 ID 最小的请求。

算法 策略 优点 局限
First Available 只检查 1 号电梯；如果 1 号电梯空闲，就把最早请求分配给它。 实现极小，适合作为最早期可运行版本。 完全忽略其他电梯，效率和公平性都较弱。
FCFS 按请求创建顺序处理，把最早请求分配给第一部空闲电梯。 请求顺序清晰，先来先服务，容易解释。 不考虑距离和方向，可能让远处电梯响应近处请求。
Nearest Idle 取最早 pending 请求，在所有空闲电梯中选择 cost 最低的电梯。 比 FCFS 多考虑空间距离，通常能减少空闲电梯的无效移动。 只考虑空闲电梯，不能给顺路运行中的电梯追加停靠。

3.3 SCAN 算法

SCAN 来源于磁盘调度中的 Elevator Algorithm，核心思想是沿一个方向持续服务，当前方向上的请求处理完后再反向。把这个思想迁移到电梯场景中，每部电梯维护一个长期扫描方向 ScanDirection，它不同于即时运动方向 Direction：Direction 表示这一 tick 是否正在上行或下行，ScanDirection 表示调度器追加请求时优先判断的服务方向。

SCAN 不只让空闲电梯接单。已有停靠计划的电梯如果“顺路”，也可以追加新的 hall 请求或 cabin 请求。顺路判断由三个函数共同完成：canAppendSCANRequest 判断是否可追加，isFloorAheadInSCAN 判断请求楼层是否在扫描方向前方，requestMatchesSCANDirection 判断 hall 请求方向是否与扫描方向一致。

SCAN 的分配过程先在“请求 × 电梯”的所有可行组合中寻找候选，再选择 cost 最低的一项。找到候选后，assignSCANRequest 会修改请求状态、记录分配 tick 和电梯 ID，并调用 insertSCANStop 把请求转换成停靠计划。停靠顺序通过 sortSCANStops 维护：扫描方向向上时楼层升序，扫描方向向下时楼层降序，因此电梯每次只需要执行 Stops[0]。

best = none
for request in pendingRequests:
    for elevator in elevators:
        if elevator is idle or request is on the scan path:
            score = EstimateAssignmentScore(system, elevator, request)
            best = lowerCost(best, request, elevator, score)
assign best request to best elevator

举例来说，如果一部电梯当前在 6 楼，ScanDirection 为 up，已有 10 楼停靠，此时 8 楼上行请求可以被追加，因为它位于当前楼层前方且方向一致；但 8 楼下行请求不会被追加，因为乘客希望下行，和这部电梯当前扫描方向不一致。这样可以避免电梯为了接一个反向请求破坏已有服务顺序。

3.4 LOOK 算法

LOOK 是 SCAN 的改进版本。传统 SCAN 会沿当前方向一直走到物理边界再反向；LOOK 则“向前看”，只要当前方向前方已经没有停靠计划，就可以提前反向，不需要空跑到顶楼或底楼。现实电梯通常也更接近 LOOK：如果上方已经没有乘客请求，就没有必要为了算法形式继续上行。

本项目中的 LOOK 复用了 SCAN 的大部分候选选择逻辑，但在判断候选前增加了 prepareLOOKDirection。该函数会检查当前扫描方向前方是否还有 stop；如果没有，而反方向还有 stop，就立即切换 ScanDirection。随后 canAppendLOOKRequest 再按更新后的方向判断新请求是否顺路。

对比项 SCAN LOOK

3.5 cost 函数

高级调度器需要在多个“请求-电梯”候选之间比较优劣，因此项目中抽出了统一的评分结构 AssignmentScore。当前 cost 公式为：Total = DistanceCost + TurnPenalty + StopPenalty - WaitCompensation。总分越低，候选越优；如果总分相同，则选择请求更早、电梯编号更小的候选，保证结果稳定并便于测试。

评分项 计算方式 设计意图
DistanceCost |currentFloor - requestFloor| * TicksPerFloor 楼层距离越远，响应成本越高。
TurnPenalty 如果需要掉头，增加 20。 避免频繁改变方向，保护已有服务顺序。
StopPenalty 已有每个 stop 增加 10。 已有任务越多的电梯越忙，不应无限追加。
WaitCompensation CurrentTick - CreatedTick 等待越久的请求获得补偿，降低长期饥饿风险。

cost 函数不单独属于某个算法，而是调度器比较候选时的共同工具。Nearest Idle 使用它在空闲电梯之间做选择；SCAN 和 LOOK 使用它在“所有 pending 请求 × 所有可行电梯”的组合中做选择。这样调度策略和评分维度分开，后续如果要加入容量、能耗或更复杂的等待惩罚，也可以集中修改评分函数。

本组数据中 SCAN 与 LOOK 指标完全相同，说明当前请求序列没有明显触发二者在“是否空跑到边界再反向”上的差异。报告中的算法选择仍保留 LOOK 的设计意义：在更稀疏或更偏向局部楼层的请求负载下，LOOK 可以在当前方向没有后续请求时提前反向，理论上比传统 SCAN 更少无效移动；而 SCAN 更适合作为经典电梯算法的基准实现。

4. Go 并发模型

为了匹配“每部电梯作为独立执行单元”的要求，在单线程同步模拟的基础上，继续使用 Go 的 goroutine、channel、select、context 和 mutex 构建并发运行模型。并发模型的目标是让系统结构更接近真实场景：HTTP 请求随时到达，后端 ticker 自动推进时间，每部电梯独立计算自己的下一步状态。

4.1 为什么需要并发

现实中的多部电梯可以同时运行，如果后端只采用单线程模拟的方式，无法体现“每部电梯独立执行”的要求以及现实电梯并发执行的特性。当前系统运行时包含三类 goroutine：net/http 为每个请求创建的 handler goroutine、自动 ticker goroutine、以及每部电梯对应的 runner goroutine。默认 5 部电梯时，系统启动后会有 5 个电梯 runner、1 个自动 tick goroutine 和若干 HTTP 请求 goroutine。

4.2 Go 并发基础

Go 的并发模型可以简单理解为：用 goroutine 表达并发执行，用 channel 表达 goroutine 之间的消息传递，用 mutex 保护必须共享的数据，用 context 结束后台任务。分别适合不同的应用场景。

4.2.1 goroutine 与线程

goroutine 由 Go 运行时调度。操作系统线程由内核调度，而 Go 运行时会把大量 goroutine 映射到较少的 OS 线程上运行，因而创建成本比操作系统线程低得多，适合把“每部电梯”“后台 ticker”“HTTP 请求处理”都表达成独立执行单元。因此在本项目中，启动 5 个电梯 goroutine 只是创建 5 个轻量任务。

4.2.2 channel 与通信式并发

channel 可以理解成 goroutine 之间传递消息的管道。一个 goroutine 通过 ch <- value 把数据发送到 channel，另一个 goroutine 通过 value := <-ch 接收数据；如果发送方或接收方暂时没有对应对象，它通常会阻塞等待。这个阻塞特性很重要，因为它天然形成同步点：发送 tick 命令后，电梯 runner 一定会在收到命令后才开始计算；Step 等待结果时，也一定会在 runner 算完后才继续合并状态。

Go 社区常用一句话概括 channel 的设计思想：不要通过共享内存来通信，而要通过通信来共享内存。它强调的是先明确数据流向，再让 goroutine 通过消息交接状态，而不是让多个 goroutine 同时读写同一份共享对象。本项目中 Step 向每部电梯 runner 发送 elevatorTickCommand，runner 返回 elevatorTickResult，这就是典型的消息传递模型。

因为 Step 与 runner 之间存在清晰的生产者和消费者关系：Step 生产一个 tick 命令，runner 消费命令并生产一个结果，因而采用 channel 来传递数据是清晰的。相比让 runner 直接修改共享的 System，channel 让数据所有权更明确：runner 拿到的是某部电梯的状态副本，返回的是新状态和完成请求 ID，真正写回共享系统仍由 Step 统一完成。

4.2.3 互斥锁与共享状态

channel 不能替代所有同步需求。System 中的 Elevators、Requests、CurrentTick 是全局共享状态，HTTP handler 和自动 tick 都可能访问它们，因此必须用 sync.Mutex 保护。也就是说，本项目使用 channel 处理“Step 与电梯 runner 的局部通信”，使用 mutex 保护“系统全局状态”。

这种组合比“全用锁”或“全用 channel”更贴合当前项目：电梯 runner 的输入输出边界很清楚，适合 channel；系统状态被 API、调度器和 Step 共同访问，适合 mutex。选择同步工具时，应当明确数据所有权和边界，从而选用恰当的方式。

4.2.4 select 与 context

select 是 Go 对多个 channel 同时等待的语法。普通接收语句只能等待一个 channel，例如只等 tick 命令；但电梯 runner 既要能接收下一次 tick，也要能在系统关闭时及时退出，所以需要同时等待多个事件。select 会在多个 case 中选择一个已经就绪的分支执行：如果收到 elevatorTickCommand，runner 就计算电梯下一状态；如果 ctx.Done() 先就绪，runner 就结束循环并退出。

context 可以理解成后台任务的生命周期控制器。创建系统时会生成一个 context，启动的电梯 runner 都持有它；系统重启或关闭时调用取消函数，ctx.Done() 对应的 channel 就会被关闭，所有 runner 都能收到退出信号。它解决的是“后台 goroutine 什么时候结束”的问题：没有 context 时，旧 runner 可能一直阻塞在 command channel 上，即使系统已经重建也不会自动消失。

因此，本项目中 select 和 context 是配套出现的：select 让 runner 能同时响应“继续工作”和“退出”两种事件，context 提供统一的退出信号。这样既能保持每部电梯独立运行，又能保证重启系统时不会留下无主的后台任务。

4.3 本项目的并发设计

本项目的并发设计如下：System 保存共享状态和两把锁；每部电梯 runner 通过 command channel 接收状态副本；runner 计算完成后通过 result channel 回传结果；最终仍由 Step 统一合并状态。设计上不允许多个 goroutine 同时随意修改同一份 System，而是把共享状态修改集中在一个合并点。

type System struct {
    mu     sync.Mutex   // 保护 Elevators、Requests、CurrentTick 等共享状态
    stepMu sync.Mutex   // 保证两个 Step 不会交错

    Elevators []Elevator
    Requests  map[int64]*Request

    elevatorCommands []chan elevatorTickCommand
}

type elevatorTickCommand struct {
    elevator Elevator   // 电梯状态副本
    done chan elevatorTickResult
}

type elevatorTickResult struct {
    elevator Elevator
    completedRequestIDs []int64
}

这段结构的核心目标很简单：每部电梯可以并行计算，但共享状态只能在受控位置修改。下面按一次 tick 的执行过程说明各部分职责。

第一步是调度和复制。Step() 先拿到 stepMu，保证不会出现两个 tick 同时推进；随后拿 mu，让调度器在一致的系统状态上分配 pending 请求。调度结束后，Step 会复制 channel 列表、深拷贝所有电梯状态，并读取本 tick 所需的只读配置。完成这些准备后释放 mu，避免后续等待 runner 期间长时间阻塞 API。

第二步是分发和并行计算。Step 把每部电梯的状态副本封装成 elevatorTickCommand，通过对应的 command channel 发给 runner。runner 收到命令后只调用 stepElevatorState 计算这一部电梯的下一状态，不读取也不写入 System。计算结束后，runner 通过 result channel 返回新电梯状态和本 tick 完成的请求 ID。

第三步是统一合并。Step 收齐所有 runner 的结果后重新拿 mu，把新电梯状态写回 s.Elevators，调用 completeRequest 完成请求、写入 SQLite 并从运行态请求表删除，最后递增 CurrentTick。这一阶段集中处理共享状态写入，因此不会出现多个 runner 同时修改同一份系统状态的问题。

电梯 runner 不直接写 System，其只接收 elevatorTickCommand 中携带的电梯状态副本，调用 stepElevatorState 计算新状态，再通过 elevatorTickResult 回传。共享状态的写回统一由 stepWithElevatorRunners 在锁内完成，这样可以把“并行计算”和“共享状态修改”分离。

这里有一个容易出错的 Go 细节：切片赋值只复制切片头，不复制底层数组。Elevator.Stops 是切片，StopPlan.RequestIDs 也是切片；如果直接把电梯结构传给 runner，副本和原对象可能共享底层数组，导致并发读写同一片内存。因此分发前需要深拷贝：

func cloneElevator(e Elevator) Elevator {
    e.Stops = append([]StopPlan(nil), e.Stops...)
    for i := range e.Stops {
        e.Stops[i].RequestIDs = append([]int64(nil), e.Stops[i].RequestIDs...)
    }
    return e
}

API handler 的并发边界也属于这套设计的一部分：GET /api/state 最终调用 Snapshot()，POST /api/request 最终调用 AddRequest()。这些方法内部会持有 mu，读取到的状态不会有误，写入请求时也不会和 Step 的合并阶段冲突。当 Step 释放 mu 等待电梯 runner 计算时，新的乘梯请求可以进入系统，并在后续 tick 被调度。

4.4 数据竞争与 go test -race

数据竞争是指两个 goroutine 同时访问同一块内存，至少一个是写操作，并且没有同步保护。并发代码即使功能测试通过，也可能因为数据竞争在高频运行时出现偶发错误。Go 提供了 race detector，可以通过 go test -race ./... 检查测试执行期间的共享内存访问是否安全。

race detector 的基本原理是在编译和运行测试时插入额外的内存访问监测逻辑。程序运行过程中，它会记录不同 goroutine 对共享变量的读写行为，并结合同步事件判断这些访问之间是否存在可靠的先后关系。如果两个访问没有被 mutex、channel、原子操作等同步手段建立顺序，而且其中至少一个是写操作，就会报告 data race。它不是静态语法检查，而是运行时检测，所以测试必须覆盖到相关并发路径才有意义。

本项目并发安全的核心原则可以概括为四点：mu 保护共享数据，stepMu 保证 tick 不交错；电梯 runner 只处理状态副本，不直接写 System；Step 中间释放 mu 以允许 API 并发访问；所有后台 goroutine 都通过 context 控制生命周期。当前报告中列出的验证命令包括 go test ./... 和 go test -race ./...。

5. 总结

本项目完成了一个可运行、可观察、可扩展的电梯调度算法可视化系统。后端使用 Go 标准库实现 HTTP API，使用统一的 System 模型维护电梯、请求、停靠计划和全局 tick；调度层实现了 First Available、FCFS、Nearest Idle、SCAN 和 LOOK；并发层用每部电梯一个 goroutine 的方式体现独立执行单元；前端使用 Vue 3 + Element Plus + Vite 展示楼层、电梯、控制面板和日志；持久化层使用 SQLite 保存完成请求，为算法实验提供数据来源。

从操作系统学习角度看，本项目把多个课程概念连接在了一起：调度算法对应请求分配策略，goroutine 和 channel 对应并发执行与通信，mutex 和 race detector 对应共享状态保护，HTTP API 和前端轮询对应系统状态的可视化观察。通过先实现同步模型、再逐步加入调度器、持久化、前端和并发模型，可以更清楚地看到工程复杂度是如何一步步产生的。

后续可改进方向主要包括三点：第一，扩展更多算法实验负载，例如午间双向流动、稀疏随机请求和突发高峰请求，并进行多轮重复实验；第二，继续完善乘客容量、上下客耗时和楼层客流模型，让模拟更接近真实电梯；第三，增强前端展示，例如增加算法轨迹回放、请求等待时间标注和历史统计面板。
