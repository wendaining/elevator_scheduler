# 电梯调度系统课程项目说明（简要版）

---

## 一、项目背景与目的

本项目来源于操作系统课程中的“进程管理”部分，旨在通过实现一个多电梯调度系统，加深对**操作系统调度机制**与**并发执行模型**的理解。

项目主要目标包括：

* 理解并实现典型调度算法（如 FCFS、SCAN 等）
* 通过电梯调度问题，类比操作系统中的进程调度过程
* 学习在特定环境下的多线程/并发编程方法 

---

## 二、项目基本要求

### 1. 系统模型

需要实现一个电梯调度系统，满足以下条件：

* 一栋大楼，共 **20 层**（但是应当扩展为可配置）
* 配备 **5 部电梯**（但是应当扩展为可配置）
* 所有电梯初始状态位于 **1 层** 

---

### 2. 并发与线程模型

系统需基于“线程思想”实现：

* 每部电梯应作为独立执行单元运行
* 多个电梯可以同时处理不同请求
* 请求产生与处理应具备一定的并发性 

---

### 3. 功能需求

#### （1）电梯内部功能

每部电梯需具备以下功能：

* 楼层选择按钮（数字键）
* 开门 / 关门按钮
* 上行 / 下行状态显示
* 当前所在楼层显示
* 报警按钮（应该就是一个类似紧急制动的按钮）

---

#### （2）楼层外部功能

每一层楼需具备：

* 上行请求按钮
* 下行请求按钮
* 电梯运行状态显示（如当前楼层、方向等）

---

#### （3）请求联动机制

系统需实现如下逻辑：

* 楼层按钮为**全局共享请求**
* 当某层的上/下行按钮被按下时，该请求应被**所有电梯感知**
* 调度系统负责决定由哪部电梯响应请求 

---

### 4. 电梯运行规则

系统应满足以下基本逻辑：

* 若电梯当前方向上无请求，则应停止运行
* 电梯仅在存在有效请求时移动
* 到达目标楼层后应执行停靠与开门操作 

---

### 5. 调度算法

* 调度策略由开发者自行设计
* 推荐实现至少一种经典算法
* 需要实现多种算法的切换

---

## 三、界面与交互要求

项目需具备基本的可视化界面，用于展示系统运行状态。

最低要求：

* 显示每部电梯的当前楼层
* 显示电梯运行方向（上/下）
* 提供楼层请求按钮

推荐增强（加分项）：

* 电梯运动动画
* 请求队列可视化
* 调度策略显示与切换
* 实时状态刷新

## 六、技术栈

结合项目特点与开发效率，采用如下技术方案：

### 后端（核心逻辑）

* 语言：Go
* 核心特性：

  * goroutine（实现电梯并发运行）
  * channel（实现请求通信）
  * net/http（提供接口）

---

### 前端（界面展示）

可选两种方案：

* 简单方案：HTML + JavaScript（推荐用于快速实现）
* 进阶方案：Vue + Element Plus

---

### 运行环境

* 使用 Linux 环境（如 WSL）进行开发
* 浏览器用于前端界面展示
* 完成项目之后使用 Docker 进行环境的打包

---

## 七、核心设计思想

本项目可以抽象为一个简化的操作系统调度模型：

| 电梯系统  | 操作系统         |
| ----- | ------------ |
| 电梯    | CPU / 资源执行单元 |
| 乘客请求  | 进程请求         |
| 调度算法  | CPU 调度策略     |
| 多电梯运行 | 多核并发         |

---

系统设计的核心在于：

* 如何分配请求给合适的电梯（调度问题）
* 如何在多电梯间协调执行（并发问题）
* 如何保证系统状态一致性（同步问题）
